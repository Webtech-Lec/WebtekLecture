Definitions

Internet

- A series of computer networks connected globally and communicate using the standard Internet Protocol Suite (TCP/IP). Any communication between two computers worldwide is possible when connected to the Internet.

- Created and tested in ARPANET

- Email was introduced (1972).

World Wide Web

- A way of retrieving and reading information shared throughout the Internet, using the HTTP/ HTTPS protocol to store and share information. Web browsers are used to view documents and files are stored in Web readable formats such as HTML and CSS.

- Created by Tim Berners-Lee in his research at Cern (1989).

Intranet

- Limited network connection within a certain region or area, within a boundary.

HTTP

HyperText Transfer Protocol

  • Created by Tim Berners-Lee
  • Jointly developed by the W3C and the IETF
  • Layered Approach
  • Plain Text Data
  • Application layer communications protocol used to access resources (hypertext/hypermedia) on the World Wide Web

Version History

  • HTTP 0.9 (1991)
  • HTTP 1.0 (RFC 1945, May 1996)
  • HTTP 1.1 (RFC 2068 January 1997, RFC 2616 June 1999), RFC 7230-7235 (June 2014)
  • HTTP 2 (RFC 7540 May 2015, patterned after SPDY)

HTTP Fundamentals

  • Runs on top of TCP/IP, using TCP port 80 by default, or TCP port 443 for HTTPS (HTTP over SSL/TLS)
  • Pre HTTP2 servers won’t volunteer data to clients. Client must request for the data before being given said data.
  • HTTP 2 follows the “push” protocol which allows servers to volunteer info based on client requests.
HTTP is based on a client-server, architecture

Clients (user agents UA): Web browsers, web crawlers/spiders, other end user tools and applications.

HTTP uses a request-response standard protocol

  • The client sends an HTTP request message to the server
  • The server processes the request and replies with an HTTP response message

HTTP is a stateless communications protocol

Servers do not keep information about clients in-between requests.

HTTP provides support for other functionalities, such as:

  • Cache Control – to cache or not to cache (Locality of Reference – only use when needed).
  • Content Media Type (MIME - Multipurpose Internet Mail Extensions) specification – no reliance on file name extension and can be opened regardless.
  • Language and character set specification (Unicode/ASCII)
  • Content/transfer Coding (byte by byte stream, compression up to 80-90%)
  • Content Negotiation – agreement between handling of content by the server and the client (file type, language, compression)
  • Client-server Protocol Negotiations (HTTP 1.0, 1.1 or 1.2)
  • Persistent Connections – client retains connection to server after request and response have been sent and received.
  • Request Pipelining – one requests at a time (wait for response).
  • Authentication/authorization

HTTP Resource Addressing

  • Uniform Resource Name (URN) – does not matter where location is; check data availability.
  • Uniform Resource Locator (URL) – same as URN but also points out the location of the data.
HTTP Resources are identified using URIs (RFC 3986) or more specifically, HTTP URLs
Scheme (HTTP or HTTPS protocols)

Ex. http:// or https://

Authority

  • domain name (resolved to an IP address using DNS) of the server where the resource resides.
  • Port number (optional)

Ex. serverhere.com:8080

Path to resource (resolved relative to the document root on the server)

  • May refer to a static or dynamic resource.

Ex. /where/here/there.pdf

Query

  • Typically provided as key=value pairs, with ampersand (&) separators between key/value pairs..
  • May be URL encoded.

Fragment identifier

Denoted via a pound sign.

Complete URL example

http://serverhere.com:8080/where/here/there.pdf?here=school&stud=dudette#heading

HTTP Request Message

CRLF – Carrier Return and Line Field (used to note the termination of a line

Request Line (CRLF - terminated line)

  • Method
  • Request URI
  • HTTP Protocol Versions

Message Headers (general, request, and/or entity leaders)

  • HTTP 1.1 requires at least the Host request header to be provided
  • Empty Line (CRLF) – done with headers
  • Message Body (payload, optional)

HTTP Response Message

Status Line (CRLF - terminated line consisting of three space-separated values)

  • HTTP Protocol Version
  • 3-Digit Status Code
  • Reason Phrase

  • Message Headers (general, request, and/or entity leaders)
  • Empty Line (CRLF)
  • Message Body (optional)

HTTP Request Methods

Methods

GET

  • Transfer a current selected representation of the resource identified by the Request URI; the retrieved resource is returned in the message of the response as an entity.
  • Most commonly used HTTP method.
  • Must be supported by all compliant general-purpose servers.

HEAD

  • Same as GET, except that the entity is not included in the response (returns only the status line and headers returned by a GET request, without the message body).
  • Used to retrieve metadata about the entity implied by the request without transferring the entity itself (tests for link validity or resource modification).
  • Like GET, must be supported by all general-purpose servers.

POST

  • Perform resource-specific processing of the entities enclosed in the message body by the resource identified by the Request URI.
  • Typically used in submitting HTML form data.
  • Has a payload.

PUT

  • Store the enclosed entity in the message body under the specified Request URI (resource identified by the Request URI is either created or replaced, using the enclosed entity).

DELETE

  • Deletes the stored enclosed entity in the message body under the specified Request URI (resource identified by the Request URI is deleted).

OPTION

  • Request information about the communication.
  • Asks the server, “What can I do?”

TRACE

  • Request a loop-back of the request message (requests the server to echo back to the client the received request message).
  • Typically used for teaching/diagnostics of the request/response shown.

CONNECT

  • Request the establishment of a tunnel to the destination origin server, and if successful, thereafter restricts its behaviour to blind-forwarding of packet, in both directions, until the tunnel is closed.
  • Commonly used to create an end-to-end virtual connection.

Safe Methods, Indempotent Methods, Cacheable Methods

  1. Safe – no modification on server side after a request (GET, HEAD, OPTIONS, TRACE).
  2. Idempotent – results stay the same after multiple requests (GET HEAD, OPTIONS, TRACE).
    • PUT – still expects new resources.
    • DELETE – no more resources to expect (remove).
  3. Cacheable – results are dependent on message headers.

Extension Methods

  • Web DAV (RFC 4918)
  • PROPFIND, PROPPATCH, MKCOL, COPY, MOVE, LOCK, UNLOCK

HTTP Message Headers

General Header Fields

  1. Cache-Control - used to specify directives to be obeyed at all times by the website caching system (determines local data storage).
  2. Connection Date - used to specify settings for website connection and establish persistent connections.
  3. Date - used to specify the date (GMT) of the request.
  4. Pragma - HTTP 1.0 general directive. Pragma: no-cache is the only directive defined in HTTP 1.0 and maintained in 1.1.
  5. Trailer - header at the end of the message instead of start.
  6. Transfer-Encoding - specifies the type of encoding done to the message to safely deliver it from the sender to the receiver.
  7. Upgrade - specifies conversion to a new protocol (http:// --> htpps://)
  8. Via - identity/Watermarking. Passes through proxy, like stamp identity, onto the next.
  9. Warning - identifier for encoding problems.

Request Header Fields

  1. Accept - used for content negotiations to be accepted.
  2. Accept-Charset - request for a specific character set (Unicode, ASCII, ANSI).
  3. Accept-Encoding - request for specific encoding (compression type).
  4. Accept-Language - request for specific language for the webpage (English, Japanese, French).
  5. Authorization - contains credentials of the request of the user-agent (authorization handshake).
  6. Expect - checks if the server can handle the request (triggers status code 100 or 417).
  7. From - email of the user-agent sending the request.
  8. Host - specifies the host and port of the requested resource. Virtualhost support. No host = bad request (400)
  9. If-Match - specifies request to be followed if the server resources matches the entity tag.
  10. If-Modified-Since - conditional request; if resource has not been modified since the given time, the resource is not returned.
  11. If-None-Match - conditional request; resource is only returned when the entity tag is matched.
  12. If-Range - specifies portions of the resource to be returned; resource is only returned when the specified range meets the request.
  13. If-Unmodified-Since - conditional request; server performs operation only if the resource has not been modified since the specified time.
  14. Max-Forwards - limits the number of proxies a trace will request go through (trace until 2nd/3rd proxy).
  15. Proxy-Authorization - authentication with a proxy as the middleman (identification of user-agent to a proxy).
  16. Range - specified partial range of content to be returned (bytes).
  17. Referrer - shows the link where the resource is from (URI).
  18. TE – trailer encoding; indicates the Transfer-Enoding to be accepted by the user agent.
  19. User-Agent - information about the sender of the request (name, compatibility, OS).

Response Header Fields

  1. Accept-Ranges - specifies the server’s acceptance of range requests for its resources.
  2. Age - informs the user-agent of how long since the response was generated by the server (how old the response is).
  3. ETag (Entity Tag) - identifier for the resource.
  4. Location - redirection of the request to the location of the resource.
  5. Proxy-Authenticate - part of the 407 status code response (user-agent is required authentication).
  6. Retry-After - part of the 503 status code response (sets time before the request can be made again).
  7. Server - server information (Apache, nginx).
  8. Vary - content negotiation; specified different sources of the entity.
  9. WWW-Authenticate - part of the 401 status code response (issues a challenge that indicates required authentication parameters).

Entity Header Fields

  1. Allow - specifies operations that can be done on an entity (PUT, POST, DELETE, etc.)
  2. Content-Encoding - indicates encoding of media type (gzip).
  3. Content-Language – indicates the intended language of the resource.
  4. Content-Length - indicates the length of the resource (bytes).
  5. Content-Location – indicates the location of the resource, where it’s from.
  6. Content-MD5(Message Digest) - hashing: change in data is a change in hash
  7. Content-Range - indicates the range of bytes; resource is sent partially to specify where it belongs in full resource body.
  8. Content-Type - indicates the extension of the resource (.html, .css, .jpg, .gif, etc.).
  9. Expires - indicates how long the user-agent can keep the content in the cache.
  10. Last-Modified - indicates when the resource was last changed.

HTTP Status Codes

The Status-Code element in a server response, is a 3-digit integer where the first digit of the Status-Code defines the class of response and the last two digits do not have any categorization role.

Informational (1xx)

It means the request has been received and the process is continuing.

  1. 100 Continue – the user agent should continue with the request.
  2. 101 Switching Protocols – can upgrade protocol then send; the server will switch protocol to those defined by the response’s upgrade header field immediately.

Success (2xx)

It means the action was successfully received, understood, and accepted.

  1. 200 Ok – most common; indicates that the user-agent request was received.
  2. 201 Created – submitted via PUT; the request has been accepted and a new resource has been created.
  3. 202 Accepted – the request has been submitted/accepted but not yet fully processed; the request may or may not be acted upon by the server.
  4. 203 Non-Authoritative Information – the meta information may not be the definitive one available from the server; it is not required and is only appropriate when the response would be 200 otherwise.
  5. 204 No Content – the request has been submitted but does not return anything; primarily used to allow for user agent input without modifying the document view (new meta data is still applied to the document, if there are any).
  6. 205 Reset Content – the request has been submitted and the server side agent is to reset the document view for the user agent.
  7. 206 Partial Content – a partial GET request for a resource has been submitted.

Redirection (3xx)

It means further action must be taken in order to complete the request.

  1. 300 Multiple Choices – there are multiple representations of the resource and the user agent is provided with a choice and is redirected accordingly.
  2. 301 Moved Permanently – the request has been moved to a new permanent URI; resource references are pointed towards the new URI.
  3. 302 Found – the resource is in a temporary URI; alteration of the location is a possibility and the user agent is alerted of such.
  4. 303 See Other – the response to a request is under a different URI; allows for the redirection of the user agent to the selected resource.
  5. 304 Not Modified – request has been submitted and accepted; document has not been modified and indicates that the resource entity is not cached.
  6. 305 Use Proxy – redirection to indicate that the resource is to be accessed through a proxy.
  7. 306 – no longer used.
  8. 307 Temporary Redirect – the client is redirected temporarily to a URI before going to the resource location.

Client Error (4xx)

It means the request contains incorrect syntax or cannot be fulfilled.

  1. 400 Bad Request – the user agent is not understood by the server; usually a URL syntax error (typos, misspellings or such).
  2. 401 Unauthorized – user agent authorization has been refused by the server due to request credentials not matching set authentication standards.
  3. 402 Payment Required – not used; reserved for future use.
  4. 403 Forbidden – user agent request is understood but not granted; the server does not wish to make the information available.
  5. 404 Not Found – a matching request URI has not been found by the server; used when no response is applicable or if the server wishes to withhold information about the request refusal.
  6. 405 Method Not Allowed – the method specified is not allowed for the requested resource.
  7. 406 Not Acceptable – the resource generates unacceptable content characteristics based off the Accept headers of the request and is not returned.
  8. 407 Proxy Authentication Required – indicates the requirement of the user agent to authenticate with the proxy before resource access.
  9. 408 Request Time-out – the user agent is unable to make a request within a set time limit by the server.
  10. 409 Conflict – the user agent request is in conflict with the resource (most likely PUT requests); only returned when the user agent is capable of resolving the conflict.
  11. 410 Gone – the requested resource is completely unavailable and no longer in the server (no redirection; no addresses to be forwarded to).
  12. 411 Length Required –the user agent failed to specify the length of the resource to be returned in accordance with the server’s Content-Length header.
  13. 412 Precondition Failed – preconditions may be set by user agents to apply to the requested resource; response code is returned if the precondition set is evaluated as false.
  14. 413 Request Entity Too Large – the requested resource is larger than what the server is expected to handle; can be the inability of the server to process a response or a set limitation of resource size.
  15. 414 Request-URI Too Long – the requested resource URI is longer than what the server is expected to handle.
  16. 415 Unsupported Media Type – the requested resource media format is unsupported by the actual resource in the server.
  17. 416 Request Range Not Satisfiable – the Accept-Range of the request is not in match with the range set by the server standards.
  18. 417 Expectation Failed – paired with the Expect request header and is returned when the server is unable to handle the request.
  19. 426 Upgrade Required – the request may be accepted by the server is the user agent switches to a different protocol.

Server Error (5xx)

It means the server failed to fulfill an apparently valid request.

  1. 500 Internal Server Error – the server has encountered an unexpected error and the user agent request is dropped.
  2. 501 Not Implemented – the server is lacking the functionality to fulfill the user agent request.
  3. 502 Bad Gateway – gateway/proxy error; invalid response from an upper connection to the server and the latter is unable to fulfill the request.
  4. 503 Service Unavailable – the server is under maintenance or overloading and is unable to process the user agent request.
  5. 504 Gateway Time-out – gateway/proxy error; untimely response from an upper connection to the server and the latter is unable to fulfill the request.
  6. 505 HTTP Version Not Supported – the server is unable to process the request due to refusal or inability to support the HTTP protocol version specified.

    Selectors
    1. Structure used as a condition in a CSS rule to determine which elements in the documentation tree are matched by the selector and are thus targeted by the formatting specified in the CSS rule declaration block.
    2. The matched elements are called the subjects of the selector
    Selector syntax
    1. Chain of one or more sequences of simple selectors separated by combinators, with one pseudo-element possibly appended to the last sequence.

      Example: div #abc> p.xyz[title]+span:first child::after

      > , + = contributor

      After = pseudo-element

      Span:first child::after = pseudo-class

    2. Sequence of simple selectors
      • Chain of simple selectors not separated by combinators
      • Always starts with a type sector or a universal selector
      • Cannot contain other type selectors or universal selectors
    3. Group of selectors
      • Coma-separated list of selectors representing the union of all elements (e.g h1, h2, h3)
